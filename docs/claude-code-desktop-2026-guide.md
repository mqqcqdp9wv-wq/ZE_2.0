# Claude Code Desktop — Полный разбор новых функций (апрель 2026)

Структурный анализ обновления Claude Code Desktop: переделанный интерфейс + Routines (автозапуск в облаке).

---

## Часть 1: Новый интерфейс Desktop

### 1.1 Параллельные сессии (Multi-Session Sidebar)

**Что это:** Боковая панель со всеми активными и архивными сессиями. Можно запускать работу по нескольким репозиториям одновременно и переключаться между ними.

**Возможности:**
- Фильтрация по статусу, проекту, окружению
- Группировка по проекту
- Автоматическая Git worktree изоляция — каждая сессия работает в своей ветке
- Когда PR мерджится или закрывается — сессия автоматически архивируется

**Как использовать:**
- Открыть боковую панель → видны все сессии
- Новая сессия → автоматически создаётся отдельный worktree
- Переключение между сессиями — мгновенное

---

### 1.2 Встроенный терминал

**Что это:** Терминал прямо внутри приложения. Можно запускать тесты, билды, git-команды — не переключаясь в отдельное окно.

- Работает параллельно с основной сессией
- Поддерживает все команды (npm, git, python и т.д.)

---

### 1.3 Встроенный файловый редактор

**Что это:** Редактор файлов внутри приложения.

- Открываешь файл → редактируешь → сохраняешь
- Подходит для быстрых правок без переключения в VS Code / другой IDE

---

### 1.4 Drag-and-Drop раскладка

**Что это:** Все панели (чат, терминал, превью, diff viewer) можно перетаскивать и размещать в любой сетке.

- Перетаскивай панели как хочешь
- Организуй рабочее пространство под свои задачи
- Несколько панелей рядом одновременно

---

### 1.5 Side Chats (Боковые чаты)

**Что это:** Можно задать вопрос «в сторону», не ломая основной поток.

**Горячая клавиша:** `⌘ + ;` (Mac) или `Ctrl + ;` (Windows/Linux)

**Как работает:**
- Side chat получает контекст из основного потока
- Но НЕ добавляет ничего обратно в основной поток
- Не мешает текущей задаче

**Пример:** работаешь над фичей, хочешь спросить "как работает этот API?" — открываешь side chat, спрашиваешь, получаешь ответ, основная задача не затронута.

---

### 1.6 Улучшенный Diff Viewer

- Полностью переписан для производительности на больших changesets
- Inline-комментарии к изменениям
- Быстрый обзор что именно Claude изменил

---

### 1.7 Preview (Превью)

- Открывает HTML-файлы прямо в приложении
- Открывает PDF-файлы
- Запускает локальные dev-серверы в preview-панели (например, Next.js)
- Видишь результат работы без переключения в браузер

---

### 1.8 Три режима отображения

| Режим | Что показывает |
|-------|---------------|
| **Verbose** | Полная прозрачность — все вызовы инструментов, шаги |
| **Normal** | Баланс между деталями и результатом |
| **Summary** | Только результаты, без промежуточных шагов |

---

### 1.9 Мониторинг Pull Request

- Следит за статусом PR
- **Auto-fix** — автоматически правит если CI упал
- **Auto-merge** — автоматически мерджит когда все проверки пройдены
- **Auto-archive** — архивирует сессию когда PR закрыт

---

### 1.10 Computer Use

- Claude может открывать приложения, кликать мышкой, управлять экраном
- Доступно на macOS и Windows
- Для Pro и Max пользователей

---

### 1.11 Dispatch (Отправка с телефона)

- Отправляешь задачу с телефона → получаешь сессию на десктопе
- Удобно когда идея пришла не за компьютером

---

### 1.12 SSH-сессии и облачные сессии

- **SSH** — подключение к удалённой машине
- **Cloud** — длительные задачи выполняются в облаке Anthropic

## Часть 2: Routines (Автозапуск в облаке)

### 2.1 Что такое Routine?

**Routine** — это сохранённая конфигурация Claude Code:
- **Промпт** (инструкция)
- **Репозиторий** (один или несколько)
- **Коннекторы** (Slack, Linear, Google Drive и т.д.)

Routine выполняется **в облаке Anthropic** — работает когда ноутбук закрыт, переживает перезагрузки.

### 2.2 Три типа триггеров

#### Schedule (По расписанию)
- Часовой, ежедневный, по рабочим дням, еженедельный
- Время в твоём часовом поясе
- Можно кастомный cron через CLI (`/schedule update`)
- Минимальный интервал: **1 час**
- Может стартовать на несколько минут позже (stagger)

#### API (По HTTP-запросу)
- Каждый routine получает свой уникальный HTTP-эндпоинт
- Bearer-токен для авторизации
- Можно передать контекст в поле `text` (например, тело алерта)
- Токен показывается один раз — сохраняй сразу

**Пример вызова:**
```bash
curl -X POST https://api.anthropic.com/v1/claude_code/routines/trig_01.../fire \
  -H "Authorization: Bearer sk-ant-oat01-xxxxx" \
  -H "anthropic-beta: experimental-cc-routine-2026-04-01" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"text": "Описание задачи или контекст алерта"}'
```

**Ответ:**
```json
{
  "type": "routine_fire",
  "claude_code_session_id": "session_01...",
  "claude_code_session_url": "https://claude.ai/code/session_01..."
}
```

#### GitHub (По событию в репозитории)
- Требует установку **Claude GitHub App** (не просто `/web-setup`)
- Поддерживаемые события:
  - **Pull Request** — opened, closed, assigned, labeled, synchronized и др.
  - **Release** — created, published, edited, deleted

**Фильтры для PR:**

| Фильтр | Что фильтрует |
|--------|---------------|
| Author | Автор PR |
| Title | Заголовок PR |
| Body | Описание PR |
| Base branch | Целевая ветка |
| Head branch | Исходная ветка |
| Labels | Лейблы |
| Is draft | Черновик или нет |
| Is merged | Смерджен или нет |
| From fork | Из форка или нет |

Операторы: equals, contains, starts with, is one of, is not one of, matches regex.

---

### 2.3 Как создать Routine

#### Через веб (claude.ai/code/routines)

1. Зайти на **claude.ai/code/routines** → **New routine**
2. Задать **имя** и **промпт** (самая важная часть — должен быть самодостаточным)
3. Выбрать **модель** (Opus, Sonnet и т.д.)
4. Добавить **репозитории** (клонируются при каждом запуске)
5. Выбрать **окружение** (Environment) — сеть, переменные, setup-скрипт
6. Настроить **триггер(ы)** — расписание / API / GitHub
7. Проверить **коннекторы** (все подключённые MCP по умолчанию)
8. Нажать **Create**
9. **Run now** — запустить немедленно для тестирования

#### Через CLI

```
/schedule          — создать новый routine
/schedule list     — список всех routines
/schedule update   — изменить routine
/schedule run      — запустить немедленно
```

Пример: `/schedule daily PR review at 9am`

> CLI создаёт только scheduled routines. Для API и GitHub триггеров — редактировать через веб.

#### Через Desktop App

**Schedule** → **New task** → **New remote task** (НЕ "New local task")

> "New local task" — это локальный scheduled task, работает на твоей машине.
> "New remote task" — это Routine, работает в облаке.

---

### 2.4 Окружения (Environments)

Каждый routine работает в окружении, которое контролирует:

| Параметр | Что делает |
|----------|-----------|
| **Network access** | Уровень доступа к интернету |
| **Environment variables** | API-ключи, токены, секреты |
| **Setup script** | Команды перед запуском (установка зависимостей) |

Есть **Default** окружение. Кастомные создаются в **Settings → Environments**.

---

### 2.5 Репозитории и ветки

- Каждый репозиторий клонируется при запуске
- Стартует с default branch (обычно `main`)
- По умолчанию Claude может пушить только в ветки `claude/*`
- Для снятия ограничения: **Allow unrestricted branch pushes**

---

### 2.6 Коннекторы (MCP)

Routine может использовать подключённые MCP-коннекторы:
- Slack (читать/писать сообщения)
- Linear (создавать задачи)
- Google Drive
- и другие

Все коннекторы включены по умолчанию — убирай лишние.

---

### 2.7 Лимиты

| План | Цена | Запусков Routines в день |
|------|------|--------------------------|
| **Pro** | $20/мес | 5 |
| **Max** | $100-200/мес | 15 |
| **Team** | корп. | 25 |
| **Enterprise** | корп. | 25 |

- Расход routine = как обычная интерактивная сессия
- При превышении лимита: если включён extra usage → продолжает на платной основе
- Без extra usage → отклоняет до сброса окна

### 2.8 Примеры использования Routines

#### 1. Обслуживание бэклога
- **Триггер:** Расписание (каждый вечер)
- **Действие:** Читает новые issues, ставит лейблы, назначает ответственных, отправляет саммари в Slack

#### 2. Тriage алертов
- **Триггер:** API (вызов от системы мониторинга)
- **Действие:** Получает stack trace, находит связь с коммитами, открывает draft PR с исправлением

#### 3. Code Review
- **Триггер:** GitHub (pull_request.opened)
- **Действие:** Применяет чеклист команды, оставляет inline-комментарии по безопасности/стилю/производительности

#### 4. Верификация деплоя
- **Триггер:** API (вызов от CD пайплайна)
- **Действие:** Запускает smoke-тесты, сканирует логи, пишет go/no-go в канал

#### 5. Актуальность документации
- **Триггер:** Расписание (еженедельно)
- **Действие:** Сканирует PR за неделю, находит устаревшую документацию, открывает PR с обновлениями

#### 6. Портирование SDK
- **Триггер:** GitHub (PR merged в одном SDK-репо)
- **Действие:** Портирует изменения в параллельный SDK на другом языке

---

### 2.9 Важные нюансы

1. **Routines выполняются от твоего имени** — коммиты, PR, сообщения в Slack идут от твоего аккаунта
2. **Нет approval-промптов** — routine работает автономно, без подтверждений
3. **Промпт должен быть самодостаточным** — routine не может задать вопрос
4. **Каждый GitHub event = отдельная сессия** (нет переиспользования сессий)
5. **Routine видна только тебе** — не расшариваются с командой
6. **Статус: Research Preview** — лимиты и API могут измениться

## Часть 4: Что важно для тебя (ze.studio)

### Routines для VK-бота

Ты можешь создать routine, которая:
1. **По расписанию** проверяет непрочитанные сообщения в ВК
2. **По GitHub триггеру** (push в main) → автоматически деплоит обновлённую структуру бота
3. **По API** → внешний сервис вызывает routine для обработки специфической задачи

### Замена ручного процесса "документ → правка → деплой → тест"

Вместо:
> Документ → правишь → просишь задеплоить → тестируешь → обратно

Можно:
1. Редактируешь `vk-bot-menu-structure.json` в Desktop
2. Коммитишь
3. Routine с GitHub триггером автоматически деплоит и проверяет

### Локальные scheduled tasks (Desktop)

Для задач, которые не нужны в облаке:
- **New task** → **New local task** в Desktop
- Работает на твоём компьютере
- Доступ к локальным файлам
- Не требует облака
